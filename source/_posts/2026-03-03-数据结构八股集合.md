# 链表与数组


## 链表已死

链表插入O(1),数组O(N),但是这个情况很难成立。因为要删除链表中间节点可能要先遍历很多节点，前提是你持有这个节点的指针，只有不断在首尾操作才是理想状态。

LRU就是一个比较典型

链表的插入删除，还可能涉及内存分配malloc(内存分配器，锁竞争，新节点cache miss等等)，可以通过内存池优化，但是远不如数组方便。

数组，业务常用vector这种，常用reserve可以很大提高性能。

    - 并发: simd等硬件并行，进一步强化数组的速度。而链表一个个节点是不连续的，必须读节点的nxt指针才能指导下一个节点，串行，依赖度高。  遍历vector : 预取器+cpu L1缓存，而 遍历 list可能是 每次指针跳转都是一次 cache miss

## 侵入式链表

侵入式链表（Intrusive Linked List）是一种**链表节点的“指针域”直接嵌在业务对象/结构体内部**的链表实现方式。也就是说：对象本身就带着“前驱/后继指针”（或 next 指针），链表不再单独分配一个“节点（node）”来包裹对象。



（linux代码可以看看，但是看不懂）

普通链表

```
struct Node{
    int data;
    Node* prev;
    Node* next;
}
```

- 节点包裹数据，数据是节点一部分，要插入数据，不可避免要新建节点，涉及内存分配。

- 对象和链表节点分离

- 把对象放入链表内，需要额外分配构造一个Node

  



侵入式链表
业务对象的数据结构之间包含链表指针。

插入数据时，只是把内部的指针改改，对象集合可能都是在某一块内存上的，新加入数据没有内存分配发生，仅仅只有几个指针的赋值，减少内存分配。链表管理的是对象中的link字段，不需要额外Node。



更灵活，性能高，但是需要你自己管理内存什么的，更复杂。

Linux内核很多这种嵌入式链表,进程调度队列，驱动设备队列，文件系统缓存等各种对象，这里的对象生命周期复杂，可能存在于多个链表，数组难以表达这种复杂归属.



```c
typedef struct ListNode {
    struct ListNode *prev, *next;
} ListNode;

typedef struct Task {
    int id;
    // ...其他业务字段...
    ListNode run_queue_link;   // 侵入式：链接字段嵌在对象里
} Task;
```

链表操作时实际上是在操作 `run_queue_link`，而不是新建一个 Node 再把 `Task*` 塞进去。

为了从 `ListNode*` 找回外层 `Task*`，一般会配合“container_of / offsetof”技巧（Linux 内核很常见）。



**侵入式链表 = 链表指针域属于对象本身，链表只是把这些指针域串起来。**



### 优点

1. **零额外节点分配** 
   不需要为每次入链额外 malloc/new 一个 node，减少内存碎片和分配开销。这在系统、实时场景，如内核，驱动，游戏引擎，网络库等要求避免频繁动态分配
2. **更好的缓存局部性** 
   链接指针和对象数据通常在同一块内存里，遍历时更可能命中 cache。
3. **一个对象可同时在多个链表里**  
   只要对象里放多个 `ListNode` 字段即可（比如同时在 run queue、wait queue）。



### 缺点

- 侵入业务结构：业务对象包含了指针，耦合更强
- 生命周期要求更严格：链表直接指向对象内部字段，对象释放前必须先从所有链表摘除，否则回形成悬空指针，野指针。
- 适用门槛高
  - 同一节点不能重复挂在同一个链表
  - 容易链表损坏
  - containerof等技巧复杂



### 适用场景

- 操作系统内核/驱动（Linux 的 `list_head` 就是经典侵入式双向链表）
- 网络库、对象池、资源管理、任务调度
- 需要高性能、低分配、低抖动（latency）的场景
- 对象本来就由框架统一管理生命周期（容易保证“先摘链再释放”）



## 无锁数据结构

### 为什么链表

lock-free wait-free数据结构，经常是用链表实现。核心是链表更新粒度小，指针操作可用CAS原子化，冲突面更可控。容易把并发修改拆成少量可重试的原子操作。

很多无锁数据结构骨架就是链表
- Treiber stack：无锁栈，本质是“头指针 + 单链表节点”
- Michael–Scott queue：无锁队列，本质是“带哨兵节点的链表 + head/tail 指针推进”
- 无锁 set/map：常见做法是“有序链表 + 标记删除 + 帮助推进（helping）”，再逐步扩展到跳表（skiplist，仍然是多层链表）

这些结构关键状态集中在几个指针上，很方便适用CAS表达和证明正确性，进度性。


#### 1. 链表关键更新改一个指针，适配CAS（或更广义的LL/SC)。

单链表很多操作的线性化点可以落在
- prev->next = new 插入
- prev->next = cur->next 删除，跳过
- head=head->next 出队出栈

这些操作用一次CAS成功即生效的方式表达，要么成功更新，要么失败重试。
相对地，数组，树，哈希表很多操作往往要维护多个位置或者多个不变量（搬运元素，旋转，重哈希），很难压缩到几个CAS里。

#### 2. 冲突局部化：只在少数节点上竞争，不必全局搬家
链表操作只影响局部相邻节点，因此并发冲突的范围小。

而基于数组的结构在扩容，rehash，resize时需要操作大量槽位，即使单步无锁，整体的协调和进度保证会更复杂，而且更容易出现大家都在同一件大事上冲突。

#### 3. 链表天生动态大小，避免扩容

无锁结构中，扩容缩容是很棘手的问题
- 需要迁移数据
- 需要让并发读写看到同一视图
- 需要保证旧表适时回收。

链表只需要按需分配节点并连接，不需要整体搬运，因此避开了这一大复杂性

#### 4. 删除更容易分阶段：逻辑删除+物理摘链
无锁里直接“真正删除并释放内存”会引出问题：别的线程可能还在读这个节点。

链表特别适合把删除拆成两步：
	1. **逻辑删除**：在指针或节点里打标记（例如把 `next` 的最低位当 mark bit，或设置 deleted flag）
	2. **物理删除**：其他线程在遍历时顺手帮你把它从链上摘掉（helping）

这样做可以让遍历继续前进，把内存回收放到后续的安全时机

#### 5. 链表无锁的的真正难点是内存回收而不是指针CAS

你会经常看到无锁链表的实现配套：

- Hazard Pointers
- Epoch/EBR/QSBR
- RCU（读多写少场景）
- 引用计数（通常成本较高）

原因是：无锁保证了结构不被锁住，但不保证节点什么时候没人用了。
链表节点频繁插删时，这个问题尤为突出。

不过它至少是“可分离的模块”，能用上述机制解决，而数组式结构的迁移/扩容把回收问题放大得更复杂。

#### 6. 对比：树和数组为何更少见（很难做）

- 树：插入删除会涉及父子多指针一致性，旋转等。线性化点更难找，步骤长，ABV/回收/帮助机制复杂
- 动态数组：扩容免费，如何安全地并发读写且无锁，复杂
- 哈希表：可以无锁（分段，渐进式扩容，版本化桶等），但扩容+迁移+回收是难点。工程上要么分段锁折中or用十分复杂的方案

#### 总结

链表能把并发更新压缩为少数指针CAS，冲突和不变量更新范围更小，避免了其他数据结构的全局搬迁和多点一致性问题。  
核心挑战是安全地内存回收



### 无锁队列Michael–Scott Queue

- 链表+哨兵节点(空队列时出入队方便)
- 原子指针：head、tail
- 节点value+原子next



#### 入队

- t=tail，n=t->next
- if n == null 则t是真的链表末尾
  - try CAS（t->next,null,newNode)
  - 这步CAS成功，通常是入队的线性化点，逻辑上元素进入队列了
  - 然后再尝试CAS（tail，t，newNode）这步失败也没事，别人会帮你推进
- elif n != null,说明tail落后了（有人插入了节点但你还没跟上）
  - try CAS(tail,t,n)
  - 然后重试插入

帮忙推进tail指针，保证lock-free进度，即使某线程挂了，其他线程也能把tail纠正回来。



#### 出队

1. 读 `h = Head`，`t = Tail`，`first = h->next`
2. 若 `first == null`：队列空，返回空
3. 若 `h == t` 但 `first != null`：说明 Tail 落后了：
   - 帮助推进 `CAS(Tail, t, first)`，重试
4. 否则：
   - 读取 `value = first->value`
   - 尝试 `CAS(Head, h, first)`  
   - **这一步 CAS 成功通常是出队的线性化点**（从逻辑上“元素被移除”发生在这里）
   - 成功后再处理旧的 `h` 的回收（注意内存回收问题）



#### 难点：ABA和内存回收

节点何时能free

- 线程A读到Head = h，准备CAS
- 线程B把h出队并free，又恰好分配到的新节点地址复用为同样的指针值。 线程A看到的还是h，但是CAS可能错误成功（ABA）

解法

- **Hazard Pointers**：线程声明“我正在读这个节点”，别人不能回收它
- **Epoch / EBR / QSBR**：用“代际”保证在某个 epoch 之前的节点不会被并发访问后再统一回收
- **Tagged pointer / 版本号**（有时可用，但不能单独解决所有回收问题）



### 无锁Map

比无锁队列更复杂，不仅有增删改查，还有扩容和并发一致性

#### 桶+每桶一个无锁链表/跳表（分离连接）

工程上最常见。每个桶都是一个条无锁有序链表（或跳表），插入也是CAS，删除是逻辑标记删除+物理摘链（helping）

- 有序链表：按key或者key hash排序，便于并发删除和帮助清理
- 标记删除：next指针低位打mark，或节点标示位
- helping：遍历到被逻辑删除的节点，帮忙移除它

##### put、remove的线性化点

- 插入新节点：`CAS(pred->next, curr, newNode)` 成功即线性化
- 删除：常拆两步  
  1) 逻辑删除：`CAS(curr->next, next, marked(next))` 线性化  
  2) 物理摘链：`CAS(pred->next, curr, next)`（帮助清理，不一定是线性化点）

##### 扩容（棘手）

一般是渐进式搬运



新表+旧表可读写但操作会顺便搬运某段范围+所有搬运完用一个原子指针切换（也可用双表并存+版本号）



- **协作式迁移（helping migration）**：任何线程在访问时如果发现 bucket 未迁移，就先迁移它
- 迁移状态用原子标志记录（例如 bucket head 指向一个特殊“已迁移”节点）



- 优点：逻辑清晰，能复用无锁链表/跳表的成熟套路
- 缺点：链表遍历成本高；扩容与内存回收仍复杂

#### 开放寻址+槽位原子状态机（高性能&复杂）

每个槽位都是一个原子cell，里面存

- key，value指针或内联值
- 槽位状态：EMPTY / OCCUPIED / TOMBSTONE / MOVED...，状态切换CAS

追求性能，缓存局部性。适合读多写少且kv可原子替换。但实现十分复杂。

#####  put/get/remove 

- `get`：线性探测/二次探测找 key；读 `cell`（通常需要 acquire 语义）验证 key 是否匹配
- `put`：找到 `EMPTY` 或 `TOMBSTONE` 的槽，CAS 抢占写入；若遇到同 key 则更新（可能需要双宽 CAS 或“指针替换”）
- `remove`：把槽位从 `OCCUPIED` CAS 成 `TOMBSTONE`（或写入墓碑标记）

##### 扩容困难

仍然是渐进扩容
- 新表建立后，旧表槽位被标记成moved并记录新表位置
- get/put/remove 发现Moved时去新表继续，并帮助搬运

难点
- 如何保证 key 不丢、不重复
- 如何避免“一个 key 同时在两张表可见”破坏线性化
- tombstone 太多会退化，需要重整（又是搬迁问题）



### 共同点

1) **CAS + 重试循环**：失败代表被别人抢先了，重读状态再试  
2) **helping**：别人卡住的“推进步骤”由我来完成，保证 lock-free 进度  
3) **安全内存回收**：Hazard Pointers / Epoch / RCU 等，否则很容易出 ABA/悬空指针
