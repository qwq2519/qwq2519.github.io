# 怎么理解LLM

llm，大语言模型，本质是根据上下文预测下一个token。



将现实世界的数据（各种文本，图片）编码为多维向量来训练

大参数：很高数量级的参数（变量），将训练集的大量数据浓缩在这些参数上。这些参数来自于预训练和微调。经过计算可以得到复杂的向量空间。

当参数达到一定规模，模型就涌现出了智能。这是大数据的力量，但llm并不能识别很多实体间的关系，这就是幻觉。

生图模型尤为明显，大模型很难识别不同实体在空间位置的关系，gemini多模态融合推理，不是单纯的图片数据，生图效果好点。



你让大模型算10个数的加减乘除，没问题。但涉及到100个，1000个呢？这时候很难计算正确了，因为llm的训练集没种长运算的数据，或者说很少。  可能用cot加强下，让llm分步计算，或者调用计算器tool。但大模型把输入的prompt传递给计算器tool也可能存在问题，格式不正确，漏了或者多了数字等。再写一个从prompt提取表达式的tool？

# tool

function call。oai提出并内化到模型里的



简单来说就是把外部函数使用json描述放在上下文中，llm在需要时返回入参和选择的工具，外部识别到再调用函数，将结果塞给llm。



tool设计原则：

- 明确工具功能，边界，精准，避免冗余，模糊，复杂
- 代码库化设计：功能单一，便于组合
- 功能正交无重叠
- 鲁棒性，错误返回友好
- 入参和返回尽量高效精简，格式，命名清晰，简单。

可以在上下文中提供每个工具调用的示例，提供给llm的工具应该是最小化，避免大量工具，并且定期根据使用率，成功率等进行优化调整，大量tool maybe可以使用rag来提供tool。



但是tool在精不在多。不然很浪费token。将所有工具，工具描述，few shot全塞进上下文，连工具的返回也一次性塞进，带来的结果自然是上下文爆炸，短对话的情况下性价比极低，有些人用cc，mcp上下文占比能有几十k。。。（cc默认窗口才200k好像）。



解法：

  - 检索：工具存入向量数据库，做相似度检索
  - 路由：用一个轻量的分类模型来筛选需要的工具
  - 分层：将工具组织为树状，分多次决策来选择具体的工具
  - 缓存：缓存历史的工具选择逻辑，记录。便于直接查询历史，直接推荐相关调用工具
  - 压缩：上下文压缩，工具描述，工具历史进行压缩
  - 摘要：用小模型对工具描述进行压缩，仅在需要时才取出完整描述
  - 动态：需要时才加载工具
  - 组合：常见工具进行组合，减少重复的描述

# agent

整合以上内容，就是ai agent。 有自主规划，反思能力，调用工具。

简单点：多问几次大模型。  高级点：llm powered workflow，围绕大模型的工作流。 工作流可以是agent自主规划也可以是人为编排的。还有多agent，更像工作流了



- 链式（DAG）
- react循环式，探索式，不断探索路径，耗时，可能死循环
- Program-Aided Language models (PAL): 让llm生成代码取执行，更灵活动态



智能体需要涉及，获取的context一般很大，所以一般不会直接全塞给它，需要实现一些记忆检索机制

- 预推理检索：推理前将所有数据塞入上下文，简单省事速度快效果差。
- 即使检索：维护轻量标识符索引（文件路径，网络链接（还可以转短链，能显著节省prompt，甚至提升回答效果）），运行时通过工具动态加载读取。节省上下文，实时性强，速度慢，需要实现相关工具和启发式规则，如claude code可以调用bash进行检索
- 混合检索：结合以上两种，需要明确这两种策略的边界。






总结： LLM powered workflow



# RAG

检索增强生成，就是给llm外挂一个知识库，也可以理解为agent的一个工具。2020年的一篇论文。随着模型能力提升，prompt里面的知识能更好的运用，rag也就火起来了。

Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks

弥补大模型预训练参数的的不足，比微调更灵活，更方便。可以提供个性化定制化记忆功能，回答更准确，节省上下文。（在大量文档资料的情况下，但是你召回要写得好很难）。

检索方式多种多样，主流方式是向量相似度检索和关键词匹配（倒排索引，全文索引什么的），图数据库的关系检索。 甚至联网搜索tool，数据库检索，文件知识库，api等都可以算是。

rag其实可以划分很多时代的，但这里不记录了），直接记录相关技术

Pre-Retrieval 优化：
- Query Rewriting（查询改写）：用大模型把模糊问题改写成检索友好格式
- Query Expansion（查询扩展）：一个问题扩展成多个角度子问题，提升召回率
- HyDE（假设文档嵌入）：先让大模型“假设”一个答案，用假设答案去检索

During Retrieval 优化：
- 混合检索（Hybrid Search）：向量检索（语义）+ BM25（关键词）并行
- Chunk 策略优化：小块检索、大块喂给 LLM
- 父文档检索（Parent Document Retrieval）：细粒度定位，粗粒度返回上下文
Post-Retrieval 优化：
- Re-ranking（重排序）：用 Cross-Encoder 精细打分，提升 Top-K 质量
- 上下文压缩（Context Compression）：剔除无关冗余，减轻 LLM 上下文压力

## 粗略实现

需要有专门的工具推理思维链：搜索-阅读-筛选-引用，逐步完成。可以形成闭环，循环、多次调用、

- 意图到查询到问题map
- 候选召回（多路检索结果）
- 证据选择与去重（排序，聚类）
- 证据整合与引用
- 迭代式计划与反思（可选，可能继续检索，纠错等）


1. 入口与编排

- 对话编排：解析意图，问题重写派生等，是否需要联网，生成搜索查询，选择工具和步骤
- 思维链：生成思维链，列出子问题，检索计划

2. 检索层（检索，数据清洗，数据入库）

- 从文档库，向量数据库，业务API等进行检索
- 检索方式：关键字检索，语义检索，混合检索，rerank等

如果是联网搜索，可能还设计网页清洗：去广告，导航，恶意脚本等，抽取正文，可能还需要反爬，限流，网站黑名单。

- 片段切分去重：窗口分块策略，minhash/simhash去重，LSH(Locality Sensitive Hashing)，合并相邻块，摘要过长段落，控制上下文长度
- 可信度评估：来源质量，时间戳，新旧对比，交叉验证

3. 生成与引用

- 将片段整合来源元数据，形成结构化prompt注入上下文，让大模型根据证据回答
- 明确引用：要在回答中标注来源，避免证据内容吞并为模型自述，便于检查
- 反思与追问：若存在证据冲突，不足，自动检索或者让向用户澄清，让其自证。一个事实最好有多源背书。

5. 缓存与性价比

- 查询缓存（结果，文本，向量）降低成本和延迟
- 命中策略：URL指纹，内容哈希，向量近邻缓存
- 限流与重试，超时回退到离线知识

### 文档分块

1. 固定长度切块

- 技术文档，代码，日志等结构规整
- 简单，快。
- 可能切断语义单元，可以前后重叠部分字符

2. 递归字符分割（结构感知）：按照优先级尝试多种分隔符进行递归切分

- 通用文本（报告，新闻，网页，md文档，html等）

- 依赖文档结构清晰度

3. 语义分块
   用NLP模型，嵌入模型，llm模型等分析语义边界，合并相似段落，拆分复杂句子，可以使用滑动窗口。特别是llm分块，可以在分块补充一些上下文，拆分命题，比如：小王在跑步。他喜欢喝水。会拆分为 小王在跑步和小王喜欢喝水等

- 长文档（手册，书籍），多模态内容（图文混合）

5.按文档结构分块（格式敏感）

### 渐进式检索/搜索

一、渐进式检索（Progressive Retrieval / Iterative Retrieval）

- 核心思想：不是一次性把所有信息取回，而是按需、分步取回。每一步都根据当前已获得的上下文，细化查询、缩小范围、拉取更相关的片段。
- 常见于：信息检索系统、向量数据库 + 大模型（RAG）场景、多回合问答的资料调取。
- 关键机制：
  1) 初始粗检：用宽泛查询拿到一批候选文档或向量相似片段。
  2) 动态细化：阅读部分结果后，基于缺口与歧义，生成更具体的子查询或新的检索键。
  3) 逐步加深：从摘要到原文、从元数据到全文、从概览到细节，逐级扩展。
  4) 停止条件：达到足够证据、置信度或资源预算用尽。
- 好处：减少无关数据、提高相关性与效率；便于在资源/时延受限的系统里做“先粗后细”。
- 风险/代价：需要更复杂的控制逻辑；如果早期检索偏了，后续也可能越走越偏。

二、渐进式搜索（Progressive Search / Iterative Search）

- 核心思想：把复杂的搜索任务拆解为一系列更小、更明确的搜索步骤，每一步根据前一步的结果调整方向和策略。
- 常见于：通用网络搜索、代码搜索、学术文献查找、问题求解/规划（甚至算法中的启发式搜索）。
- 关键机制：
  1) 任务分解：把总问题拆为可验证的子问题或假设。
  2) 策略迭代：替换或细化关键词、切换数据源、改变筛选条件与排序标准。
  3) 证据累积：边搜边验证，逐步构建对答案的“证据链”。
  4) 反馈闭环：利用用户反馈或自动评估指标来决定下一步搜索方向。
- 好处：更适合不明确、开放式的问题；能逐步校正偏差，发现更优信息源。
- 风险/代价：多轮交互增加时延；需要良好的“何时停止”的判断。

三、两者关系与区别

- 相同点：
  - 都是“迭代-反馈-再行动”的流程。
  - 都强调从粗到细、从少量高价值信息出发，逐渐完善答案。
- 不同点（侧重点）：
  - 渐进式检索更偏“取数策略”：如何分层次地从数据库/索引/向量库中拉取更相关的文档片段。
  - 渐进式搜索更偏“任务求解流程”：如何通过多步搜索与策略调整来逼近正确答案或高质量来源。
  - 技术实现上，渐进式检索常与检索器、索引结构、分层召回、再排序（rerank）紧密相关；渐进式搜索则更强调查询重写、来源切换、评估与决策。

- 数据主要在可控索引或向量库里，且你能做精排/重排：优先渐进式检索。
- 问题开放、信息分散在不同来源、需要不断调整策略：优先渐进式搜索。
- 实务中两者常结合：搜索阶段先确定方向与来源，检索阶段在确定的数据域里分层获取与验证。

四、典型应用示例

- RAG问答：
  - 渐进式检索：先用问题做向量检索得摘要段落；模型读后发现需要具体年份，再发起二次检索限定年份与实体；最后取原文段验证事实。
  - 渐进式搜索：模型先搜概念定义，再搜权威来源（标准/白皮书），接着搜最新进展与反例，最终生成带引用的综述。
- 企业知识库：
  - 渐进式检索：先查知识类别与目录，再下钻到特定文档与章节，再取需用的小段落。
- 开发者调试：
  - 渐进式搜索：先搜错误码含义，再限定到框架版本与平台，随后搜索同类 issue 与补丁提交，最后确认复现条件与修复方案。

## 图RAG
就是基于图数据库的rag。抽取实体关系，建立知识图谱

适用于：复杂因果/依赖/关系推理、多跳问答、可解释性要求高、知识庞杂且动态。但是文档要删除新增很麻烦，也有一定局限性。

不适用于：简单检索问答，关系稀疏。数据少且文档，极高实时性要求。否则徒增复杂度和资源消耗


## RAG vs long context
长上下文优势是无需检索，链路简单。但成本高，受限于注意力机制，无法准确回答，很难管理文档，更新困难延迟高
RAG：强溯源需求，频繁更新，成本低点，适合大规模知识库。

# 知识库化

将个人的知识进行记忆化持久化。

1. 整合：在特定时机（session结束或者用户显式指令）时，调用llm提取用户偏好，并进行结构化（非结构化数据可以保留便于溯源）
2. 索引：加各种tag，可以建立关联索引（类似图结构），给多模态数据加附加文本。  图结构，时间线结构，增加摘要等元信息等，便于入库
3. 更新：系统自动更新 or 用户主动更新 ，可以添加溯源链
4. 遗忘：过期遗忘 & 长时间降低权重 & 用户主动遗忘
5. 检索：向量相似度检索 & 多模态检索 & 上下文关键字检索，总之整合多源召回
6. 压缩：历史记忆压缩 & 检索结果压缩 ，可以使用向量形式压缩AutoCompressor和基于llm自然语言压缩MemGPT。
