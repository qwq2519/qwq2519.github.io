# introduction

## what

互联网，云连接起来的服务器，客户端，数据中心（数据中心本身也可能是分布式系统），即很多个计算机连起来

## why

- 连接物理隔离的机器，便于用户共享数据，各种资源的共享
- 并行，提高容量，性能 ---mapreduce
- 容错，容故障，高可用

## mapReduce

早期谷歌在构建搜索引擎索引时，需要大量计算，很多人都要写分布式系统应用。造这个论文就是为了便于编写分布式应用，为了容错性

不是一个通用的库，而是需要使用特定的方式编写，

- 开发者编写map和reduce函数，这些函数都是函数式的，无状态的。顺序编写
- 交给mapReduce框架，框架负责在多个机器上计算

## Abstract View

- map是无状态的，各自处理输入，输出结果。------高吞吐，扩展
- shuffle阶段：对相同数据进行reduce，不同reduce也是独立运行的，进行聚合

经典就是多文件单词计数，排序（桶排）

google用了GFS。 输入你要处理的文件，然后还有一个master作为协调器，负责对worker进行调度，worker（失败，没用响应）也要对其重启（重新执行map/reduce）实现容错。所有map都完成后再进行reduce

一般来说，input 存在哪个机器，对应的map就会上面运行worker，然后本地存储结果（中间结果），告诉协调器，中间结果后面会发送到reduce机器，reduce机器最后在输出到GFS中

一个map/reduce可能运行多次，worker1发生了网络分区，导致worker2也执行了这个map，但函数式运行两次也没事，是无状态的，确定性的，相同输入必须有相同输出



其他故障

- 协调器故障？目前的实现是这样认为的，一般认为不会，当协调器故障，所以任务都要重跑，因为协调器是有状态的，要对其实现容错很复杂。尽管有些技术可以给协调器实现容错，但是协调器一般运行在单个机器上，不会认为其会发生故障
- 某worker出现问题，很慢。没问题，它的任务可能会被其他快速机器执行，重复执行没事，慢速worker相当于做了备份任务。性能瓶颈从慢worker---》被replication的快worker

## GFS 

目标：高性能，因此采用了复制和容错，一致性比较差

下面介绍的GFS有一部分不符合学术论文设计的

- master服务器只有一个，没复制
- 不一致


 ### 特点

- 大，容量大
- 性能高，自动分片，并行读取
- 全局：共享，方便
- 容错性：高容错，自动容错



### 设计

为大型计算设计的分布式文件管理系统

有个master来知道文件在哪里，客户端定时和mast通信来检索信息

一个大文件可能分为多个chunk存在不同服务器，客户端通过与master通信后知道文件chunk位置，直接和这些chunk服务器通信。多个客户端可以读取一个文件的不同chunk，提升IO性能。 master会返回你需要的文件chunk列表或者特定的chunk的位置，一个chunk可能被多个服务器存储



#### master

维护以下状态信息

- 文件名 ： chunks handler数组

chunk handler ： version和存储该chunk副本的服务器列表，租约到期时间

- 一个chunk通常是有多个副本，一主两从。

- 日志和检查点：重要信息，写入磁盘后再响应客户端。而上面两个一般放在内存。日志相当于redis的AOF日志和RDB快照。master定期会在符合一致性状态下生成检查点



为了一致性，master能正常允许，哪些需要持久化存储，或者只放在内存就行？

- chunks handler数组，可以通过日志来获取，而日志是持久化的，所以这个也是需要持久化的
- chunk handler到服务器等： 论文写了，master启动时要求服务器告诉它们自己有什么。 因此这个是易失性的，不需要持久化。但是版本号必须持久化才知道服务器中的有没有过期，判断拥有最新chunk的服务器启动了吗。版本号很重要，master和chunk server都写在磁盘上

#### Read File

- 客户端发送文件名和offset给master
- master找到对应的chunk handle，回复客户端对应的chunk服务器和版本号
- 客户端缓存此列表（减少通信次数，提升性能，并且master只有一个，因此为了提升性能肯定要缓存）
- 客户端选择最近的服务器读取chunk
- chunk server检查版本号没问题就返回。这个就是缺陷，但是GFS尽量做到完美

#### Write File

追加记录到文件，这个操作很常见很重要，因为mapreduce主要就是追加，而不是随机修改

- 客户端通信，确定写入位置
- master根据文件名找到对应chunk handle
- 通过chunk handle找到chunk服务器列表
  - 假设没主服务器，即这是第一次：递增版本号，然后选择主从服务器（primary和secondary），master授予主服务器租约（会过期）。然后master再将版本号写入磁盘并响应客户端。客户端会联系发送真正的数据给最近的一个secondary，每个secondary向下一个secondary传递数据，pipline模型。但是各个chunk服务器并未真正的持久化数据，下一步让客户端发送版本号，primary服务器会检查版本号是否一致，不一致则不允许写入，还有primary会检查自身租约是否有效（过期的话集群可能存在另一个primary），然后才是选择一个偏移量，写入磁盘，primary再通知其他secondary节点使用offset进行持久化。最后再响应客户端，如果其中一个服务器没有正确写入就会报错，然后就是客户端重试。一般至少重试一次，重试再选择的offset就不一定和上一次一样了。因此你可能会发现有些chunk服务器会有重复的数据！这和正常的文件系统不一样。 客户端要严格遵守primary返回的offset，而不是和文件系统交互，一般来说会有给库执行插入id执行保证数据唯一性。 这样设计的原因之一可能是当某个服务器断开连接还能继续写入。



#### consistency

上面的提到，客户端会缓存chunk server list，如果chunk集群里面发生了网络分区，客户端仍然可能会读到独立出来的旧的chunk数据。这是没办法的。

但是版本号不会递增，只有重新选主才会递增版本号。写入操作提到的是序列号，是写入顺序，权限。

primary需要写入数据时，会从master获取其他secondary，这时发现有secondary宕机了，这时候会怎么样呢（视频说论文模糊了）。。。但是可能还会有频繁的心跳检测，或者租约到期重新选主



primary和master断开了，master要重新选主也必须等到primary租约到期，因为可能仍然存在一些客户端这这个断开的primary通信。不然的画会脑裂，这时候写入会很奇怪





如何获取强一致性？

更新所有p-s节点或者全不更新

GFS是专门为mapreduce定制的，还有其他不同一致性要求的文件系统



# Primary/Backup replication

FMT(the virtual memory or VMware fault tolerant)

## Failures

讨论下有哪些故障，复制能解决哪些，不能解决哪些，我们期望通过复制方案能做什么

- fail-stop failures:发生故障，计算机停止运行。最常见的，很多系统论文都只解决这个，它假设计算机在正常工作的情况下没有故障，一有故障就停止。计算机系统，软件都尽可能将其他故障转换为这种故障
- logic bugs、configuration error，mailcious erros：比如除以0，备份系统也可能执行一样的错误操作，复制操作没办法解决这种故障

这门课都在讨论第一种，假设只有第一种，如何处理停机故障。看不同主备方案的作用



## challenges

DMFT论文

- 发生故障，primary服务器发生故障了吗？分布式系统中，网络分区和机器故障无法区分，必须采取措施避免脑裂：客户端的一个子集和一个primary通信，而客户端的另一个子集和另一个primary通信
- 如何让主备服务器同步，实现理想的主从切换（即完全同步），对客户端透明？必须要保证服务器处理的顺序，执行相同操作的结果必须相同
- 故障转移：主备切换时，必须确保没有其他主在运行。比如primary要响应给客户端，但这时宕机了，主从切换后的新主需要判断是否需要重新发送
- 谁来执行故障转移，新主怎么选

要保证服务器处于确定性的状态，要保证主从服务器处于相同的状态

## Two Approachs

- state transfer：primary服务器定时生成一个检查点发送给备服务器。为了同步，primary每次响应客户端之前都要都要传输给备服务器
- replicated state machine：primary服务器直接发送 操作给备服务器

都是让primary和backup服务器处于一致状态

第二种更常见，也有混合方案

复制状态机要求所有操作都是确定性的，什么是非确定性指令呢，比如获取时间，必须将每个非确定性指令转换为确定性指令。



## level of operation to replicated

- 应用程序级别的操作 file append，需要修改应用程序来是实现复制状态机

- 机器级别，寄存器级别，硬件级别，状态是寄存器，内存，操作是x86指令。通过机器级别指令复制，不需要改操作系统和应用程序，能实现容错，透明复制

  中断指令

  传统的机器复制，需要购买硬件，需要大量硬件机制

  但是论文提到，可以不使用纯粹硬件（航空航天领域会用，硬件投票等），而是虚拟机，虚拟化，VMFT，对应用程序透明。非常强的一致性，但是该方案不支持多核

## overview

虚拟机监视器将一个物理硬件虚拟为多台，上面再跑多个操作系统，这样操作系统中断不会直接发送给硬件，外部中断也不会直接发送给操作系统。  不同机器之间通过虚拟机监视器的日志通道进行同步，primary发送到backup物理机。

此外还有独立的存储服务器，不仅给服务器提供存储功能，还提供一个块用于发生故障时进行仲裁选举主节点，发生网络分区时将根据存储服务器的仲裁块进行选主。服务器将发送数据包，在存储服务器的仲裁块上cas标志为1，抢占设置1，抢到的服务器来作为新主节点。



出现故障时，人工接入，让新主虚拟机生成镜像副本（并发送到backup节点）并重置上面的仲裁块标志。生成镜像期间不响应客户端



什么会导致主从状态不同步？ 分歧来源于非确定性指令，如获取时间指令，如在指令流中输入数据和定时器中断指令位置不一样，要控制中断（并发本质是切换线程的中断）。     本论文放弃多核，不然要完全同步的话就需要很复杂，确保两个机器同一个线程都争取到了锁，这很复杂。后来的论文好像做了多核

其实主从节点执行的指令流都是一样的，但同一时刻都在执行同一个，只是执行顺序一样，因此确定性指令不需要做通信（都有二进制文件副本），而在非确定的指令执行时需要做特殊处理，比如中断

主节点收到中断后，通过日志通道发送让备节点知道多少条指令后执行xx中断，因此备份节点需要落后通道上一个消息，落后一个轮次，类似于批量处理。 因为我们无法判断什么时候有中断。

比如获取时间指令，当执行到这条指令时，会被标记为无效指令，控制权从操作系统转移到虚拟机监视器（trap)。备份服务器会判断非确定性指令会修改哪个寄存器，然后等待主服务器发送来该寄存器的值



故障转移，上面提到的一个场景，主服务器响应客户端前宕机了，但是数据包已经通过日志通道发送给b节点了，b节点重新选主后会响应客户端吗：不会，因为主服务器没到达发送数据包的状态，b节点也不可能到达，因此最后是等待客户端重试

但是如果没成功从日志通道发送呢？这就是不一致性了，因此有一个一致性规则：输出数据前要保证发送给b服务器的所有消息得到确认，这也可能导致外部发现我们内部系统不是单机统一的。

客户端也可能收到重复的响应，这是可以接受的，网络容错模型就假设网络本身会复制消息，TCP等协议也接受

目前这个论文支持是1p1b

因此任何分布式系统都有一些对外部输出响应的规则，就是为了一致性，对外部透明



这就是经典的复制状态机，特别是非确定性指令，输出规则

性能瓶颈，性能下降大就是主服务器必须等待从服务器响应，特别是因为中断导致频繁通信降低性能。这也是人们选择应用程序级别的复制状态机而不是硬件级别，但这需要修改应用程序
