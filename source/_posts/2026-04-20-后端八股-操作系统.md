# 操作系统

## 进程与线程

都是操作系进行任务调度和资源管理的抽象，但是侧重点不同

- 进程是资源分配的基本单位
- 线程是CPU调度的基本单位

### 定义
进程：一个正在运行的程序实例，操作系统会给它分配独立的虚拟地址空间，文件描述符，页表，信号处理环境，权限等资源

线程：进程内的一个执行流，一个进程可以有多个线程，这些线程共享进程大部分资源，如代码段，堆，打开的文件，socket等。但每个线程都有自己的程序计数器，寄存器上下文，栈空间

### 原理

为什么要用线程和进程这个模型？
- 隔离：不同任务之间希望互不影响，所以引入进程，提供地址空间隔离和权限隔离
- 并发：同一个任务内部希望有多个执行流并发，提高CPU利用率，响应速度，所以引入线程

为什么有进程了还要线程？
- 为了提高同一应用内的并发处理能力，比如一个web服务既要处理请求，又要做日志刷盘，超时回收，后台任务等，每个都用进程，成本高。
- 进程间共享数据困难，通信成本高

线程：同一进程内多个任务并发执行，共享数据方便，创建销毁成本的低。web服务，数据库，中间件就会大量使用线程，协程模型

为什么还要进程呢？
- 某个线程内存越界，非法操作等，可能导致整个进程崩溃
- 安全边界弱，不同模块不好隔离
- 垃圾回收，依赖冲突，资源泄露互相影响

不同服务一般用进程隔离，进程内部再用线程提高并发


进程通常有
- 独立虚拟内存空间
- 页表
- 打开的文件描述符表
- 信号处理上下文
- 当前工作目录
- 用户态/内核态相关资源
- 各种内核管理信息，如PID

线程通常有
- 独立线程栈
- 程序计数器PC
- 寄存器上下文
- 线程本地存储TLS
- 线程状态信息

线程之间共享内容一般有
- 进程地址空间
- 全局变量
- 堆内存
- 代码段
- 打开的文件/socket
- 进程级别的信号处理设置

### 切换都在切什么

本质是保存了当前执行现场，恢复下一个执行现场



进程重在地址空间切换

因此进程切换开销一般包含

- 线程上下文

- 保存和恢复CPU上下文
- 切换内核栈
- 切换虚拟地址空间
- 可能导致TLB失效
- CPU cache命中率下降
- 页表基址等
- 安全上下文，资源边界等待

核心在于虚拟地址空间变了，TLB之前的地址映射会失效，CPUcahce 被破坏



线程切换开销
- 保存和恢复线程上下文
- 调度开销，调度状态
- cache miss
- 内核中的task、thread control block
- 同进程内不同线程共享虚拟地址空间，页表，代码段、堆，文件描述符，因此不需要切换整个地址空间，轻得多



CPU上下文包含什么，一个执行流要在cpu上跑，至少需要以下状态

- 程序计数器PC：下一条指令
- 通用寄存器：当前计算过程的中间值
- 栈指针SP：当前栈位置
- 状态寄存器：条件码，中断状态等
- 可能还要浮点寄存器，simd寄存器

切换就是保存当前任务的这些状态，从下一个任务的上下文中恢复，然后CPU从下一个任务的PC上继续执行



上下文切换多了会导致

- CPU忙于恢复上下文，而不是处理请求，没业务价值（CPU很高但是QPS不涨）
- cache命中率下降（不同线程工作不一样）
- 调度器压力大，就绪队列长，线程频繁抢占，调度开销大
- 线程过多也会导致锁竞争激烈



怎么排查，用指令咯

- vmstat ：看上下文切换次数，运行队列
- pidstat:看进程是否自愿切换
- top ：看哪个线程CPU高
- perf “看热点是否存在锁竞争，调度，系统调用
- java用jstack看线程状态
- go用pprof看goroutine，muntx，block profile

线程多，锁竞争严重，频繁系统调用，IO阻塞太多，线程池配置不合理，GC或runtine调度压力大



### 核心区别

1. 资源分配和隔离性不同
- 进程有独立地址空间，天然隔离，一个进程崩溃不会影响另一个进程崩溃。浏览器内核就是采用多进程模型，标签页崩溃不会影响其他标签页
- 线程共享进程内存，一个线程写坏内存，可能会拖垮整个进程

2. 创建和销毁成本不同
- 进程创建成本高，需要分配独立的地址空间，建立页表，复制或继承父进程资源
- 线程创建轻量，主要是分配栈和初始化上下文

3. 切换开销不同
- 进程切换通常更重，因为涉及地址空间切换，页表切换，TLB刷新等
- 线程切换比较轻，特别是同进程内切换，不需要切换整个地址空间。但也不少很便宜，如果线程过多，调度开销，缓存失效也很明显

4. 通信方式不同
- 进程间通信需要IPC机制，比如管道，消息队列，共享内存，socket，信号，mmap等
- 线程间通信直接通过共享内存就行了，但是必须处理同步问题，比如锁，信号量，条件变量，cas

5. 稳定性不同
- 多进程模型隔离性好，更适合高可靠场景
- 多线程模型通信方便，性能高，但是容易出现竞态，死锁，内存可见性问题

### fork、exec、wait

#### fork
创建子进程

特点
- 子进程几乎是父进程的副本，会复制父进程的执行上下文，地址空间描述，文件描述符等
- 父子进程从fork返回后继续往下执行
- 返回值不同，父进程fork返回值是子进程pid，子进程拿到0

现代操作系统的fork通常不是立刻全量拷贝，而是写时复制cow copy on wrie：fork后父子进程先共享同一份物理页，页表标记为只读。谁写了，谁就触发缺页中断，内核再进行复制。

好处就是创建子进程很快，如果子进程马上exec，很多页不需要复制。大大降低了初始复制成本，成本在后面的进程管理，地址空间切换，后续写入的页复制


#### exec
不少创建新进程，而是让当前进程加载并执行一个新程序

执行后：
- 当前进程的代码段，数据段，堆，栈等用户态内容会被新程序替换
- PID通常不变
- 打开的文件描述符默认会继承，除非设置close-on-exec

经典模式就是先fork，让子进程再exec，父进程继续管理子进程，shell命令就是这套逻辑，这样十分灵活，子进程里调整标准输入输出，环境变量，权限，工作目录等

#### wait
父进程用wait/waitpid回收子进程退出状态

如果子进程退出但是父进程没回收。就会变成僵尸进程
- 僵尸进程不运行了
- 但内核还保留pid，退出码等信息，等待父进程读取。这会占用很多内存，CPU等资源，还要PID，进程表项。
- 大量僵尸进程可能导致系统无法创建新进程

如果父进程先退出，子进程就会变成孤儿进程，通常被init/systemd接管


### 调度角度

操作系统真正调度到CPU上运行的是线程，也就是说一个进程是否运行，本质是看它里面的线程有没有被调度到

线程常见状态
- 就绪：等待CPU
- 运行：正在占用CPU
- 阻塞/等待：等锁，等IO，等事件
- 终止：运行结束

CPU密集型任务：多线程未必越多越好，线程数通常接近于CPU核数
IO密集型：可以适当增加线程数，因为大部分线程时间都花在阻塞等待上

### 进程间通信IPC

可以分为同机和跨机

- 管道pipe:半双工，适合父子进程简单通信，内核维护的一块缓冲区，生命周期随进程
- 命名管道FIFO：允许无亲缘关系的进程通信，在文件系统中有一个名称。
- 消息队列：内核维护的消息链表，按消息收发。允许进程以结构化消息形式交换数据（消息有类型和优先级），消息有边界，不像管道是字节流。
- 共享内存：最快，因为不需要频繁拷贝，适合大量数据交换，但是需要自己控制同步。就是允许多个进程将同一块物理内存映射到它们各自的虚拟地址空间。
- 信号量：主要用于同步和互斥，不是传输大量数据的。本质是一个计数器，同步原语，支持原子操作P和V。
- 信号 signal：通知型，适合时间通知，不适合复杂数据传输。异步通知进程发生某个事件（软件中断，预定义多种信号），进程可以用kill()发送信号。
- socket：最通用，跨机也能用，通过文件系统，IP，端口都可以，可以是流式(类似TCP)或数据报式(类似UDP)。
- mmap：通过映射文件实现共享

通常是共享内存最快，因为减少了数据拷贝，但是开发复杂度最高，需要自己处理并发同步

### 线程同步方式

线程共享内存，最大的问题就是并发安全

常见同步机制
- 互斥锁mutex：最常用，保护临界区
- 读写锁 rwlock：读多写少场景适用
- 自旋锁：锁持有时间很短时可以减少线程切换
- 条件变量： condition variable：线程间等待/通知
- 信号量 semaphore：控制并发资源数量
- 原子操作CAS：适合轻量无锁状态
- 屏障 barrier：多个线程相互等待到某个点再继续

常见问题
- 死锁：多个线程在互相等待资源
- 活锁：都在重试，但是谁也不往前走
- 饥饿：某个线程长期抢不到资源
- 竞态条件：执行顺序不确定导致结果错误
- 内存可见性：一个线程修改了变量，另一个线程看不到

### 用户线程和内核线程
- 内核线程：操作系统内核直接调度，真正参与CPU调度的实体
- 用户线程：用户态运行时管理，内核不直接感知
- 协程：本质是更轻量的用户态执行单元，栈通常可伸缩，扩展，非常适合高并发IO场景。但这不是银弹，如果频繁调用阻塞系统调用，可能会拖住线程，或者CPU密集型任务，也解决不了CPU瓶颈。 调试，栈追踪，上下文传播更复杂

常见模型
- 1:1 实现简单，主流系统常用，比如Linux的pthread
- N:1 切换快，但无法利用到多核，一个阻塞可能导致拖住全部
- N:M 调度灵活，但实现复杂

java传统线程基本可以当成1:1，go的goroutine是典型的M：N，GPM模型本质是在用户态做调度，降低线程切换成本

### 多线程/多进程

多进程适合
- 服务隔离，比如不同的微服务独立部署
- 高可靠场景：Nginx master-worker，redis的单进程模型，多worker进程互相隔离
- 利用多核时按照进程维度进行隔离

更关注隔离性，稳定性，比如高风险插件执行，不同租户隔离，崩溃域控制

多线程适合
- 单服务并发处理请求
- 异步任务，定时任务，日志刷盘，连接管理等
- 数据库连接池，业务线程池，各种池

更关注通信效率和资源共享，比如并发任务处理，低延迟协作，共享内存数据

- java web服务：一个JVM进程，内部有大量业务线程，GC线程，IO线程
- go服务：一个进程，大量goroutine，底层映射到少量线程
- MySQL：一个mysqld进程，内部多个线程处理连接，IO，刷盘，后台清理等
- Nginx:多进程事件驱动模型，每个worker单线程处理大量连接

### 其他问题
- 线程安全问题：多个线程同时修改共享变量，没用加锁或者原子保护
- 死锁：锁顺序不一致，或者持锁调用外部系统
- 线程池配置不合理：线程数多导致频繁切换，队列过长导致延迟堆积
- 进程数量过多，内存占用大，调度开销大
- fork使用不当：比如多线程程序fork后只保留调用fork的线程，容易出现锁状态异常
- 共享内存滥用：容易出现复杂并发bug
- 线程泄露：线程没用正常退出，导致资源长期占用
- 僵尸进程：子进程退出后，父进程不回收
- 孤儿进程：父进程退出后，子进程被init/systemd接管
- 一个线程阻塞拖垮全局：线程池被慢请求占满，导致整个服务不可用

## IO模型

指的是用户线程发起一次读写，数据怎么从内核到用户态，线程是阻塞，非阻塞，还是由内核/框架通知你。

### 定义

是操作系统对读写操作的处理方式，主要关注
- 等待数据准备好，比如网卡收到数据，内核socket buffer里有数据可读
- 把数据从内核空间拷贝到用户空间

不同模型区别，本质上
- 这两个阶段谁来等
- 线程会不会阻塞
- 内核会不会通知
- 一次能不能管理多个连接

### 原理

#### 1. 阻塞IO
经典模型，比如调用Read()，数据没到，线程就一直阻塞直到数据到了。内核再把数据拷贝到用户缓存区后，read()返回

特点
- 等待数据阶段阻塞
- 数据拷贝阶段阻塞
- 编程简单，一连接一线程

适用于：连接数不高，早期服务器模型，业务逻辑简单
但线程利用率低，高并发下线程数暴涨，线程切换成本高，线程池容易打满

#### 2. 非阻塞IO
把socket设置为no-blocking，调用read()如果没数据，会直接返回错误EAGAIN等，需要应用自己轮询重试

特点
- 等待数据阶段不阻塞
- 但需要用户线程不停询问
- 拷贝数据阶段通常还是同步

适用：单个fd控制，但实际用的不多
如果一直轮询，会导致CPU空转，编程复杂度高，不适合管理海量连接

一般搭配IO多路复用

#### 3. IO多路复用
典型接口是select poll epoll

原理
- 用户线程先把多个fd注册到内核
- 然后调用select poll epoll_wait阻塞等待
- 当某个fd可读，可写、异常时，内核就返回就绪事件
- 应用在对这些就绪fd执行read/write

- 多路复用不是内核帮你读数据，而是内核告诉你哪个fd就绪了
- 真正的数据拷贝，通常还是调用read(),还是同步的IO

select
- 用位图表示自己要监听fd，有最大fd数限制
- 调用时把整个fd集合从用户态拷贝到内核态，内核遍历所有fd，看那些就绪了。返回后，用户态还需要再遍历一次，看哪些fd就绪了。
- 重复传递，线性扫描，连接很多但活跃很少，效率低

poll
- 对select的改进
- 用链表/数组描述fd，没select的fd数量限制
- 但本质还是线性扫描，仍然慢

epoll

关键优化
1. 监听集合和等待动作的分离（监听集合管理）
- 通过epoll_ctl把fd注册到内核的红黑树或者类似的集合
- 后续调用epoll_wait,不需要再传递整个fd集合
- 注册一次，后续可以多次等待

2. 内核维护就绪队列（就绪事件获取）
- 某个fd就绪，内核把它放入readylist，epoll_wait返回的是就绪队列
- 不需要全量扫描所有fd。关注就绪事件，减少数据拷贝和扫描事件

epoll实例会维护一个红黑树，每个被监听的fd注册后会对应一个红黑树节点，记录了这个fd相关信息。epoll_ctl各种操作本质是监听这个树。
当某个fd发生了用户关心的事件，内核会直接把这个fd对应的节点放到就绪队列，这样epoll_wait调用时只需要检查就绪队列即可。
还有个等待队列，会放阻塞的线程等待唤醒。

适用于各自高并发web服务器，因为高并发服务的特点是连接多但同一时刻活跃的不多

支持LT和ET两种触发方式

- LT水平触发：只要缓冲区还有数据，即fd还处于可读或者可写状态就会一直通知。比如socket还有数据没读完，下次epoll_wait就会继续返回这个事件。编程简单，但可能重复通知，事件次数更多。
- ET边缘触发：状态变化时通知一次，需要一次性把数据读到EAGAIN。效率高，但更容易写错，漏读可能会卡住连接。
- ET模型下，fd必须非阻塞，写和读都要用循环来实现。

写事件不能一直注册，否则容易空转。长连接要多关注fd上限，连接泄露，内存占用等

问题
- 只解决等的问题，不代表业务处理一定高效。
- 如果单线程事件循环里有耗时操作，可能阻塞住整个reactor
- ET模型，半包拆包，连接关闭处理等更复杂


#### 4. 信号驱动IO
应用先注册一个信号处理函数，内核在fd就绪时会发送信号通知应用。

理论上可以减少主动轮询，但实际用的少。信号机制本身复杂，可重入，安全性，调试都不方便。

#### 5. 异步IO
真正的异步，应用发起aio_read()立即返回，内核负责等待数据，拷贝数据，全部完成后通知应用。

多路复用是通知你可以读了，应用才自己去处理。异步IO则是通知你已经读完了，直接处理结果。 

高性能磁盘IO和特定异步框架。

但生态不是很好，编程模型更复杂。网络编程用多路复用比较多。

#### 总结
- BIO：TOmcat早期阻塞模型，简单RPC服务
- 多路复用：Redis单线程网络模型，Nginx，Netty Reactor
- GO：开发者写起来像是同步，但runtime底层会把网络fd挂到epoll上做调度，这是同步写法+异步事件驱动的封装
- JAVA NIO：Selector本质是多路复用
- 高性能存储：会关注AIO，io_uring

#### 细节

同步/异步和阻塞非阻塞不少一个概念
- 阻塞/非阻塞：关注调用线程会不会挂起，没数据直接睡眠等待/立刻返回错误。
- 同步/异步：关注结果是不是调用方主动获取，谁来拷贝数据，还是内核完成后通知

比如epoll+read
- 等待事件仍然可以阻塞在epoll_wait。fd也可以是no-blocking
- 但数据读取仍然是同步的

多路复用不是银弹：如果业务逻辑很重单reactor会成为瓶颈。所以一般是reactor+线程池或者主从reactor多线程模型。

### Reactor和Proactor

IO模型常见延申

#### Reactor
应用先等事件，事件来了再自己处理读写。
比如用epoll_wait得到可读写事件，应用再用read、write。

多路复用常见模型都是reacttor，redis，nginx，netty，go的网络库等

#### Proactor
应用先发起异步IO，内核把整个IO完成后通知应用，应用只处理数据。接近于真正的异步IO。

### 零拷贝
IO模型关注谁等，谁通知，线程组不阻塞。零拷贝关注数据搬运链路怎么优化，减少cpu copy工作。两者是可以相结合。
